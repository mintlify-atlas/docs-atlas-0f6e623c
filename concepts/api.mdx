When you create a new Better Auth instance, it provides you with an `api` object. This object exposes every endpoint that exists in your Better Auth instance. And you can use this to interact with Better Auth server side.

Any endpoint added to Better Auth, whether from plugins or the core, will be accessible through the `api` object.

## Calling API Endpoints on the Server

To call an API endpoint on the server, import your `auth` instance and call the endpoint using the `api` object.

```ts title="server.ts"
import { betterAuth } from "better-auth";
import { headers } from "next/headers";

export const auth = betterAuth({
    //...
})

// calling get session on the server
await auth.api.getSession({
    headers: await headers() // some endpoints might require headers
})
```

### Body, Headers, Query

Unlike the client, the server needs the values to be passed as an object with the key `body` for the body, `headers` for the headers, and `query` for query parameters.

```ts title="server.ts"
import { auth } from "@/lib/auth"

await auth.api.getSession({
    headers: await headers() // headers containing the user's session token
})

await auth.api.signInEmail({
    body: {
        email: "john@doe.com",
        password: "password"
    },
    headers: await headers() // optional but would be useful to get the user IP, user agent, etc.
})

await auth.api.verifyEmail({
    query: {
        token: "my_token"
    }
})
```

<Note>
Better Auth API endpoints are built on top of [better-call](https://github.com/bekacru/better-call), a tiny web framework that lets you call REST API endpoints as if they were regular functions and allows us to easily infer client types from the server.
</Note>

## Getting Headers and Response Object

When you invoke an API endpoint on the server, it will return a standard JavaScript object or array directly as it's just a regular function call.

But there are times when you might want to get the `headers` or the `Response` object instead. For example, if you need to get the cookies or the headers.

### Getting Headers

To get the `headers`, you can pass the `returnHeaders` option to the endpoint.

```ts title="server.ts"
import { auth } from "@/lib/auth"

const { headers, response } = await auth.api.signUpEmail({
	returnHeaders: true,
	body: {
		email: "john@doe.com",
		password: "password",
		name: "John Doe",
	},
});
```

The `headers` will be a `Headers` object, which you can use to get the cookies or the headers.

```ts
const cookies = headers.get("set-cookie");
const customHeader = headers.get("x-custom-header");
```

### Getting Response Object

To get the `Response` object, you can pass the `asResponse` option to the endpoint.

```ts title="server.ts"
import { auth } from "@/lib/auth"

const response = await auth.api.signInEmail({
    body: {
        email: "",
        password: ""
    },
    asResponse: true
})
```

## Error Handling

When you call an API endpoint on the server, it will throw an error if the request fails. You can catch the error and handle it as you see fit. The error instance is an instance of `APIError`.

```ts title="server.ts"
import { auth } from "@/lib/auth"
import { APIError, isAPIError } from "better-auth/api";

try {
    await auth.api.signInEmail({
        body: {
            email: "",
            password: ""
        }
    })
} catch (error) {
    if (isAPIError(error)) {
        console.log(error.message, error.status)
    }
}
```

## Architecture

Better Auth's API architecture is built on a modular design that allows for extensibility and type safety.

### Endpoint Creation

Endpoints are created using the `createAuthEndpoint` function which wraps Better Call's `createEndpoint`. This provides:

- **Type-safe route definitions**: Endpoints are defined with full TypeScript support
- **Automatic validation**: Request body, query, and headers are validated
- **Context injection**: Each endpoint receives an `AuthContext` with database, session info, and more

```typescript
import { createAuthEndpoint } from "better-auth/api";
import { z } from "zod";

const myEndpoint = createAuthEndpoint(
  "/my-endpoint",
  {
    method: "POST",
    body: z.object({
      name: z.string(),
    }),
  },
  async (ctx) => {
    // ctx.context provides auth-specific context
    // ctx.body is validated and typed
    return ctx.json({ success: true });
  }
);
```

### Context Object

Every endpoint handler receives a context object with useful properties:

- `options`: The Better Auth configuration
- `db`: Kysely database instance
- `adapter`: ORM-like database methods
- `internalAdapter`: Internal helper functions (e.g., `createSession`, `createUser`)
- `session`: Session configuration (expiresIn, updateAge)
- `secret`: The secret key for signing/encryption
- `baseURL`: The base URL of the auth server
- `logger`: Logger instance
- `tables`: Core table definitions

### Router System

Better Auth uses Better Call's router system to organize endpoints:

```typescript
import { createRouter } from "better-call";

const router = createRouter();

router.get("/user", getUserEndpoint);
router.post("/user", createUserEndpoint);
```

The router automatically handles:
- Path matching and parameters
- Method filtering (GET, POST, etc.)
- Middleware execution
- Error handling

### Middleware

Middleware can be added at different levels:

1. **Global middleware**: Runs on all requests
2. **Route-specific middleware**: Targets specific endpoints
3. **Plugin middleware**: Added by plugins for specific functionality

```typescript
import { createAuthMiddleware } from "better-auth/api";

const authMiddleware = createAuthMiddleware(async (ctx) => {
  // Check authentication
  const session = await ctx.context.internalAdapter.getSession(token);
  
  if (!session) {
    throw new APIError("UNAUTHORIZED");
  }
  
  return {
    context: {
      ...ctx,
      session,
    },
  };
});
```

### Type Inference

Better Auth provides full type inference from server to client:

```typescript
// Server
export const auth = betterAuth({
  // ... config
});

// Client automatically infers all endpoints and types
import { createAuthClient } from "better-auth/client";

const client = createAuthClient();
// client.signIn.email is fully typed
```

## Built-in Endpoints

Better Auth provides a comprehensive set of built-in endpoints:

### Authentication
- `POST /sign-up/email` - Email/password sign up
- `POST /sign-in/email` - Email/password sign in
- `POST /sign-in/social` - OAuth sign in
- `GET /callback/:provider` - OAuth callback handler
- `POST /sign-out` - Sign out current session

### Session Management
- `GET /get-session` - Get current session
- `GET /list-sessions` - List all user sessions
- `POST /revoke-session` - Revoke a specific session
- `POST /revoke-other-sessions` - Revoke all other sessions
- `POST /revoke-sessions` - Revoke all sessions

### User Management
- `POST /update-user` - Update user information
- `POST /delete-user` - Delete user account
- `POST /change-email` - Change user email
- `POST /change-password` - Change user password
- `POST /set-password` - Set password for OAuth users

### Account Management
- `GET /list-accounts` - List linked accounts
- `POST /link-social` - Link a social account
- `POST /unlink-account` - Unlink an account

### Email Verification
- `POST /send-verification-email` - Send verification email
- `GET /verify-email` - Verify email with token

### Password Reset
- `POST /request-password-reset` - Request password reset
- `POST /reset-password` - Reset password with token

### Tokens
- `POST /refresh-token` - Refresh access token
- `GET /get-access-token` - Get current access token

### Utility
- `GET /ok` - Health check endpoint
- `GET /error` - Error endpoint for testing

<Tip>
All endpoints follow REST conventions and return consistent response formats. Check the individual endpoint documentation for detailed request/response schemas.
</Tip>
