Better Auth is designed to be type-safe. Both the client and server are built with TypeScript, allowing you to easily infer types.

## TypeScript Config

### Strict Mode

Better Auth is designed to work with TypeScript's strict mode. We recommend enabling strict mode in your TypeScript config file:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "strict": true
  }
}
```

if you can't set `strict` to `true`, you can enable `strictNullChecks`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "strictNullChecks": true,
  }
}
```

<Warning>
If you're running into issues with TypeScript inference exceeding maximum length the compiler will serialize, then please make sure you're following the instructions above, as well as ensuring that both `declaration` and `composite` are not enabled.
</Warning>

## Inferring Types

Both the client SDK and the server offer types that can be inferred using the `$Infer` property. Plugins can extend base types like `User` and `Session`, and you can use `$Infer` to infer these types. Additionally, plugins can provide extra types that can also be inferred through `$Infer`.

### Client-Side Type Inference

```ts title="auth-client.ts" 
import { createAuthClient } from "better-auth/client"

const authClient = createAuthClient()

export type Session = typeof authClient.$Infer.Session
```

The `Session` type includes both `session` and `user` properties. The user property represents the user object type, and the `session` property represents the `session` object type.

### Server-Side Type Inference

You can also infer types on the server side.

```ts title="auth.ts" 
import { betterAuth } from "better-auth"
import Database from "better-sqlite3"

export const auth = betterAuth({
    database: new Database("database.db")
})

type Session = typeof auth.$Infer.Session
type User = typeof auth.$Infer.Session.user
```

## Additional Fields

Better Auth allows you to add additional fields to the user and session objects. All additional fields are properly inferred and available on the server and client side.

```ts title="auth.ts"
import { betterAuth } from "better-auth"
import Database from "better-sqlite3"

export const auth = betterAuth({
    database: new Database("database.db"),
    user: {
       additionalFields: {
          role: {
              type: "string",
              input: false
            } 
        }
    }
   
})

type Session = typeof auth.$Infer.Session
// Session.user.role is now typed as string
```

In the example above, we added a `role` field to the user object. This field is now available on the `Session` type.

### The `input` property

The `input` property in an additional field configuration determines whether the field should be included in the user input. This property defaults to `true`, meaning the field will be part of the user input during operations like registration.

To prevent a field from being part of the user input, you must explicitly set `input: false`:

```ts
additionalFields: {
    role: {
        type: "string",
        input: false
    }
}
```

When `input` is set to `false`, the field will be excluded from user input, preventing users from passing a value for it.

By default, additional fields are included in the user input, which can lead to security vulnerabilities if not handled carefully. For fields that should not be set by the user, like a `role`, it is crucial to set `input: false` in the configuration.

## Inferring Additional Fields on Client

To make sure proper type inference for additional fields on the client side, you need to inform the client about these fields. There are two approaches to achieve this, depending on your project structure:

### 1. For Monorepo or Single-Project Setups

If your server and client code reside in the same project, you can use the `inferAdditionalFields` plugin to automatically infer the additional fields from your server configuration.

```ts title="auth-client.ts"
import { inferAdditionalFields } from "better-auth/client/plugins";
import { createAuthClient } from "better-auth/client";
import type { auth } from "@/lib/auth"; // Import the auth instance as a type

export const authClient = createAuthClient({
  plugins: [inferAdditionalFields<typeof auth>()],
});
```

### 2. For Separate Client-Server Projects

If your client and server are in separate projects, you'll need to manually specify the additional fields when creating the auth client.

```ts title="auth-client.ts"
import { createAuthClient } from "better-auth/client";
import { inferAdditionalFields } from "better-auth/client/plugins";

export const authClient = createAuthClient({
  plugins: [inferAdditionalFields({
      user: {
        role: {
          type: "string"
        }
      }
  })],
});
```

## Type Inference Architecture

Better Auth's type system is built on advanced TypeScript features to provide seamless type safety:

### Plugin Type Augmentation

Plugins can extend core types using TypeScript's module augmentation:

```typescript
import type { BetterAuthPlugin } from "better-auth";

const twoFactorPlugin = () => {
  return {
    id: "two-factor",
    schema: {
      user: {
        fields: {
          twoFactorEnabled: {
            type: "boolean",
          },
        },
      },
    },
  } satisfies BetterAuthPlugin;
};

// The twoFactorEnabled field is now inferred in User types
```

### Cross-Framework Type Safety

Types are shared across server and all client frameworks:

```typescript
// Server
export const auth = betterAuth({
  user: {
    additionalFields: {
      preferences: { type: "string" },
    },
  },
});

// React Client
import { createAuthClient } from "better-auth/react";
import type { auth } from "@/lib/auth";

const client = createAuthClient({
  plugins: [inferAdditionalFields<typeof auth>()],
});

const session = client.useSession();
// session.data.user.preferences is typed as string
```

### Union Type Resolution

Better Auth uses advanced TypeScript utilities to resolve union types from plugins:

```typescript
import type { UnionToIntersection } from "better-auth/types";

// Combines all plugin types into a single type
type CombinedUser = UnionToIntersection<
  | { role: string }
  | { twoFactorEnabled: boolean }
>;
// Result: { role: string; twoFactorEnabled: boolean }
```

## Type Utility Functions

Better Auth provides several utility types:

### Infer Plugin Types

```typescript
import { InferPlugin } from "better-auth/client";
import type { myPlugin } from "./my-plugin";

const client = createAuthClient({
  plugins: [InferPlugin<typeof myPlugin>()],
});
```

### Infer Auth Configuration

```typescript
import { InferAuth } from "better-auth/client";
import type { auth } from "@/lib/auth";

type AuthConfig = InferAuth<typeof auth>;
```

## Database Schema Types

Better Auth generates TypeScript types from your database schema:

```typescript
import type { DBFieldAttribute } from "better-auth/types";

const userSchema = {
  fields: {
    email: {
      type: "string",
      required: true,
      unique: true,
    } satisfies DBFieldAttribute,
  },
};
```

## Best Practices

### 1. Always Use Type Inference

```ts
// Good - types are inferred
const session = authClient.useSession();
if (session.data) {
  console.log(session.data.user.email);
}

// Avoid - manual typing can lead to inconsistencies
const session: { user: { email: string } } = authClient.useSession();
```

### 2. Use Type Guards for Optional Fields

```ts
import type { Session } from "better-auth/types";

function requiresRole(session: Session | null): session is Session {
  return !!session && 'role' in session.user;
}

if (requiresRole(session)) {
  // session.user.role is safely accessible here
  console.log(session.user.role);
}
```

### 3. Leverage Plugin Types

```ts
import { organizationClient } from "better-auth/client/plugins";
import type { auth } from "@/lib/auth";

const client = createAuthClient({
  plugins: [organizationClient<typeof auth>()],
});

// Organization types are now inferred
const orgs = await client.organization.list();
// orgs is properly typed
```

### 4. Export Types for Reuse

```ts title="types.ts"
import type { auth } from "@/lib/auth";
import type { authClient } from "@/lib/auth-client";

export type Session = typeof authClient.$Infer.Session;
export type User = typeof authClient.$Infer.Session.user;
export type AuthOptions = typeof auth.options;
```

Then use them across your application:

```ts title="components/user-profile.tsx"
import type { User } from "@/types";

interface UserProfileProps {
  user: User;
}

export function UserProfile({ user }: UserProfileProps) {
  // user is properly typed
  return <div>{user.name}</div>;
}
```

## Common Type Issues

### Issue 1: Type Inference Exceeding Maximum Length

**Problem**: TypeScript compiler error about type inference being too deep.

**Solution**: Ensure `declaration` and `composite` are disabled in your `tsconfig.json`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "declaration": false,
    "composite": false,
    "strict": true
  }
}
```

### Issue 2: Additional Fields Not Inferred

**Problem**: Custom fields don't appear in type hints.

**Solution**: Use the `inferAdditionalFields` plugin:

```ts title="auth-client.ts"
import { inferAdditionalFields } from "better-auth/client/plugins";
import type { auth } from "@/lib/auth";

const client = createAuthClient({
  plugins: [inferAdditionalFields<typeof auth>()],
});
```

### Issue 3: Plugin Types Not Merging

**Problem**: Multiple plugins' types aren't combining correctly.

**Solution**: Ensure all plugins are properly typed and added to both server and client:

```ts title="auth.ts"
import { betterAuth } from "better-auth";
import { twoFactor, organization } from "better-auth/plugins";

export const auth = betterAuth({
  plugins: [twoFactor(), organization()],
});
```

```ts title="auth-client.ts"
import { twoFactorClient, organizationClient } from "better-auth/client/plugins";
import type { auth } from "@/lib/auth";

const client = createAuthClient({
  plugins: [
    twoFactorClient<typeof auth>(),
    organizationClient<typeof auth>(),
  ],
});
```

<Tip>
Better Auth's type system is designed to provide automatic type inference. In most cases, you shouldn't need to manually define types - let TypeScript infer them from your configuration.
</Tip>
