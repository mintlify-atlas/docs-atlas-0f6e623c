## Overview

Better Auth provides a robust session management system with support for cookie-based sessions, session refresh, and optional cookie caching for improved performance.

## Session Lifecycle

A session in Better Auth follows this lifecycle:

<Steps>

### Creation

Sessions are created when a user signs in:

```typescript
const session = await ctx.context.internalAdapter.createSession(
  userId,
  {
    ipAddress: getIp(request),
    userAgent: request.headers.get("user-agent")
  }
);
```

### Storage

Sessions are stored in:
1. **Database**: Primary storage for session records
2. **Cookies**: Session token stored in HTTP-only cookies
3. **Secondary Storage** (optional): Redis/memory for fast lookups

### Validation

On each request, sessions are validated:
- Token is verified from cookie
- Session is fetched from database or cache
- Expiration is checked
- Session is refreshed if needed

### Refresh

Sessions are automatically refreshed based on `updateAge`:

```typescript
const auth = betterAuth({
  session: {
    expiresIn: 60 * 60 * 24 * 7, // 7 days
    updateAge: 60 * 60 * 24, // refresh after 24 hours
  }
});
```

### Revocation

Sessions can be revoked:
- Single session: `/revoke-session`
- All sessions: `/revoke-sessions`
- Other sessions: `/revoke-other-sessions`

</Steps>

## Session Configuration

### Basic Options

```typescript
const auth = betterAuth({
  session: {
    // Session expiration time in seconds (default: 7 days)
    expiresIn: 60 * 60 * 24 * 7,
    
    // How often to refresh session in seconds (default: 24 hours)
    updateAge: 60 * 60 * 24,
    
    // How long a session is considered "fresh" for sensitive operations
    freshAge: 60 * 60, // 1 hour
    
    // Disable automatic session refresh
    disableSessionRefresh: false,
    
    // Defer session refresh to reduce database writes
    deferSessionRefresh: false
  }
});
```

### Cookie Cache

Enable cookie caching to reduce database queries:

```typescript
const auth = betterAuth({
  session: {
    cookieCache: {
      enabled: true,
      maxAge: 60 * 5, // 5 minutes
      strategy: "compact", // or "jwt" or "jwe"
      refreshCache: {
        enabled: true,
        updateAge: 60 // refresh after 60 seconds
      },
      version: "1" // invalidate cache when version changes
    }
  }
});
```

**Source:** `/packages/better-auth/src/api/routes/session.ts:112`

### Caching Strategies

<Tabs>
  <Tab title="Compact">
    ```typescript
    // Default: Uses HMAC-signed JSON
    strategy: "compact"
    ```
    
    - Smallest size
    - HMAC signature for integrity
    - Base64URL encoded
  </Tab>
  
  <Tab title="JWT">
    ```typescript
    // JWT format (signed but not encrypted)
    strategy: "jwt"
    ```
    
    - Standard JWT format
    - Readable payload (base64 decode)
    - HMAC signature
  </Tab>
  
  <Tab title="JWE">
    ```typescript
    // JWE format (encrypted)
    strategy: "jwe"
    ```
    
    - Fully encrypted payload
    - Most secure option
    - Slightly larger size
  </Tab>
</Tabs>

## Session Cookies

Better Auth uses two types of cookies:

### Session Token Cookie

Stores the session token:

```typescript
{
  name: "better-auth.session_token",
  httpOnly: true,
  secure: true, // in production
  sameSite: "lax",
  path: "/",
  maxAge: 604800 // 7 days
}
```

### Session Data Cookie

Caches session data when cookie cache is enabled:

```typescript
{
  name: "better-auth.session_data",
  httpOnly: true,
  secure: true,
  sameSite: "lax",
  path: "/",
  maxAge: 300 // 5 minutes (default)
}
```

**Source:** `/packages/better-auth/src/cookies/session-store.ts`

### Cookie Chunking

When session data exceeds 4KB, Better Auth automatically splits it into multiple cookies:

```typescript
// Automatically handled
better-auth.session_data.0
better-auth.session_data.1
better-auth.session_data.2
```

This is based on browser cookie size limits (4096 bytes).

**Source:** `/packages/better-auth/src/cookies/session-store.ts:9`

## Session Retrieval

### Get Current Session

Retrieve the current user's session:

```typescript
// In an endpoint
import { sessionMiddleware } from "better-auth";

const endpoint = createAuthEndpoint(
  "/protected",
  {
    method: "GET",
    use: [sessionMiddleware]
  },
  async (ctx) => {
    const { session, user } = ctx.context.session;
    
    return ctx.json({
      userId: user.id,
      email: user.email
    });
  }
);
```

### Session Middleware

Better Auth provides several middleware options:

#### sessionMiddleware

Requires a valid session:

```typescript
use: [sessionMiddleware]
```

#### sensitiveSessionMiddleware

Requires a valid session and bypasses cookie cache:

```typescript
use: [sensitiveSessionMiddleware]
// Always checks database, ignoring cached session
```

Use for sensitive operations like password changes or account deletion.

#### freshSessionMiddleware

Requires a session that was recently authenticated:

```typescript
use: [freshSessionMiddleware]
// Session must be within freshAge
```

#### requestOnlySessionMiddleware

Requires session for client requests, but not server calls:

```typescript
use: [requestOnlySessionMiddleware]
```

## Session Refresh

### Automatic Refresh

Sessions are automatically refreshed when:

1. **Time-based**: Session age exceeds `updateAge`
2. **Request-based**: User makes a request
3. **Cookie cache**: Cookie cache expires

### Refresh Logic

The refresh calculation:

```typescript
const sessionIsDueToBeUpdatedDate = 
  session.expiresAt.valueOf() - expiresIn * 1000 + updateAge * 1000;

const shouldBeUpdated = sessionIsDueToBeUpdatedDate <= Date.now();
```

For example, with a 7-day expiry and 24-hour update age:
- Session created: Day 0
- First refresh: Day 6 (7 days - 24 hours = 6 days)
- Subsequent refreshes: Every 24 hours

**Source:** `/packages/better-auth/src/api/routes/session.ts:388`

### Deferred Refresh

Reduce database writes by deferring refresh:

```typescript
const auth = betterAuth({
  session: {
    deferSessionRefresh: true
  }
});
```

With deferred refresh:
1. **GET /get-session**: Returns session with `needsRefresh` flag
2. **POST /get-session**: Performs the actual refresh

This allows the client to decide when to refresh:

```typescript
const { data } = await client.getSession();

if (data?.needsRefresh) {
  // Refresh in the background
  client.refreshSession();
}
```

**Source:** `/packages/better-auth/src/api/routes/session.ts:414`

## Session Management Endpoints

### GET /get-session

Retrieve current session:

```typescript
const response = await fetch("/api/auth/get-session", {
  credentials: "include"
});

const data = await response.json();
// { session: Session, user: User } | null
```

Query parameters:
- `disableCookieCache`: Bypass cookie cache
- `disableRefresh`: Skip session refresh

### POST /get-session

Refresh session (when `deferSessionRefresh` is enabled):

```typescript
const response = await fetch("/api/auth/get-session", {
  method: "POST",
  credentials: "include"
});
```

### GET /list-sessions

List all active sessions:

```typescript
const response = await fetch("/api/auth/list-sessions", {
  credentials: "include"
});

const sessions = await response.json();
// Session[]
```

### POST /revoke-session

Revoke a specific session:

```typescript
await fetch("/api/auth/revoke-session", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ token: sessionToken }),
  credentials: "include"
});
```

### POST /revoke-sessions

Revoke all user sessions:

```typescript
await fetch("/api/auth/revoke-sessions", {
  method: "POST",
  credentials: "include"
});
```

### POST /revoke-other-sessions

Revoke all sessions except the current one:

```typescript
await fetch("/api/auth/revoke-other-sessions", {
  method: "POST",
  credentials: "include"
});
```

## Multi-Session Support

Enable multiple concurrent sessions per user:

```typescript
import { multiSession } from "better-auth/plugins";

const auth = betterAuth({
  plugins: [
    multiSession({
      maximumSessions: 5 // limit concurrent sessions
    })
  ]
});
```

Without this plugin, creating a new session invalidates previous sessions.

## Session Storage Patterns

### Database-Only

Simplest approach - sessions stored only in database:

```typescript
const auth = betterAuth({
  database: /* your database */,
  session: {
    cookieCache: { enabled: false }
  }
});
```

**Pros:**
- Simple
- Immediate revocation
- Single source of truth

**Cons:**
- Database query on every request
- Higher latency

### Database + Cookie Cache

Faster with cookie caching:

```typescript
const auth = betterAuth({
  database: /* your database */,
  session: {
    cookieCache: {
      enabled: true,
      maxAge: 300 // 5 minutes
    }
  }
});
```

**Pros:**
- Faster response times
- Reduced database load
- Automatic cache refresh

**Cons:**
- Revocation has delay (up to maxAge)
- Larger cookie size

### Database + Secondary Storage

Optimal performance with Redis:

```typescript
const auth = betterAuth({
  database: /* your database */,
  secondaryStorage: {
    get: (key) => redis.get(key),
    set: (key, value, ttl) => redis.set(key, value, { EX: ttl }),
    delete: (key) => redis.del(key)
  },
  session: {
    cookieCache: { enabled: true }
  }
});
```

**Pros:**
- Fast session lookups
- Reduced database load
- Near-immediate revocation
- Scalable

**Cons:**
- Additional infrastructure (Redis)
- More complex setup

## Stateless Sessions

For completely stateless authentication without a database:

```typescript
const auth = betterAuth({
  database: undefined, // no database
  session: {
    cookieCache: {
      enabled: true,
      strategy: "jwe", // encrypted
      maxAge: 60 * 60 * 24 * 7,
      refreshCache: true // enable stateless refresh
    }
  },
  account: {
    storeStateStrategy: "cookie" // OAuth state in cookies
  }
});
```

<Warning>
Stateless mode is suitable for simple use cases but has limitations:
- No session revocation (except expiry)
- No session listing
- Limited to cookie size constraints
- OAuth state in cookies (size limits)
</Warning>

## Session Security

### Don't Remember Me

Offer "don't remember me" option:

```typescript
// On sign-in
await auth.api.signInEmail({
  email: "user@example.com",
  password: "password",
  dontRememberMe: true
});
```

This prevents automatic session refresh, effectively creating a session-only cookie.

### Session Fingerprinting

Track sessions by IP and user agent:

```typescript
const session = await createSession(userId, {
  ipAddress: getIp(request),
  userAgent: request.headers.get("user-agent")
});
```

Use this data to detect suspicious activity or implement device management.

### Session Hijacking Prevention

1. **HTTP-Only Cookies**: Session tokens are not accessible via JavaScript
2. **Secure Flag**: Cookies only sent over HTTPS in production
3. **SameSite**: CSRF protection via SameSite=Lax
4. **Token Rotation**: Tokens change on refresh
5. **Short Expiry**: Combined with regular refresh
