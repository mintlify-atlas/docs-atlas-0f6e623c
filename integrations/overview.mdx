## Common Patterns

All framework integrations follow these common patterns:

### Server Setup

1. **Create Auth Instance**: Configure Better Auth with your database and providers
2. **Mount Handler**: Add the auth handler to your API routes
3. **Add Plugins**: Use framework-specific plugins for cookie handling

### Client Setup

1. **Create Client**: Initialize the auth client with framework-specific imports
2. **Use Hooks/Stores**: Access session data using reactive primitives
3. **Call Actions**: Use built-in methods for sign in, sign out, and sign up

## Choosing Your Integration

<AccordionGroup>

<Accordion title="Which framework should I choose?">
  Choose the framework you're already using for your project. Better Auth is designed to work seamlessly with your existing setup. If you're starting fresh:
  
  - **Next.js**: Best for React-based full-stack apps with SSR
  - **SvelteKit**: Great for performance-focused apps with smaller bundles
  - **SolidStart**: Ideal for reactive apps with fine-grained updates
  - **Astro**: Perfect for content-heavy sites with minimal JS
  - **Nuxt**: Excellent choice for Vue developers
</Accordion>

<Accordion title="Can I use Better Auth with multiple frameworks?">
  Yes! Better Auth is framework-agnostic. You can use the same auth server with multiple client frameworks. For example, you could have a Next.js web app and a React Native mobile app both authenticating against the same Better Auth server.
</Accordion>

<Accordion title="What about other frameworks?">
  Better Auth can work with any JavaScript framework through the vanilla client. For frameworks not listed here, check out the [Express integration](/integrations/overview) or [Other Frameworks](/integrations/overview) guide.
</Accordion>

</AccordionGroup>

## Framework-Specific Features

Each integration provides framework-specific utilities:

- **Route Handlers**: Convert Better Auth handlers to framework-native formats
- **Cookie Plugins**: Automatically set cookies in server actions and API routes
- **Session Helpers**: Access session data in server and client code
- **Type Safety**: Full TypeScript support with inferred types
- **SSR Support**: Server-side rendering for protected pages
