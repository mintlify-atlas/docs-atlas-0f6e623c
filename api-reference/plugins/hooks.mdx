---
title: 'Hooks'
description: 'Plugin hooks for intercepting and modifying requests and responses'
---

Hooks allow plugins to run code before or after specific endpoints are called. Unlike middleware, hooks run even when endpoints are invoked directly (not just through HTTP requests).

## Hook Configuration

Hooks are defined in the `hooks` property of a plugin:

```ts
hooks: {
  before: [
    {
      matcher: (context) => boolean,
      handler: AuthMiddleware
    }
  ],
  after: [
    {
      matcher: (context) => boolean,
      handler: AuthMiddleware
    }
  ]
}
```

## Before Hooks

Before hooks run before the endpoint handler executes. They can:
- Validate requests
- Modify the context
- Short-circuit execution by returning a response
- Add custom logic before endpoint execution

### Example: Validate Headers

```ts
before: [{
  matcher: (context) => {
    return context.path === "/sign-up/email"
  },
  handler: createAuthMiddleware(async (ctx) => {
    const customHeader = ctx.request?.headers.get("x-custom-header")
    
    if (!customHeader) {
      throw new APIError("BAD_REQUEST", {
        message: "Missing required header"
      })
    }
    
    return {
      context: ctx
    }
  })
}]
```

### Example: Add Custom Context

```ts
before: [{
  matcher: (context) => {
    return context.path?.startsWith("/api/auth/")
  },
  handler: createAuthMiddleware(async (ctx) => {
    // Add custom data to context
    return {
      context: {
        ...ctx,
        customData: {
          timestamp: Date.now(),
          requestId: generateId()
        }
      }
    }
  })
}]
```

## After Hooks

After hooks run after the endpoint handler completes. They can:
- Modify the response
- Log data
- Trigger side effects
- Clean up resources

### Example: Log Successful Sign-ins

```ts
after: [{
  matcher: (context) => {
    return context.path === "/sign-in/email"
  },
  handler: createAuthMiddleware(async (ctx) => {
    const response = ctx.context.returned
    
    if (response && typeof response === 'object' && 'user' in response) {
      console.log(`User ${response.user.id} signed in`)
      
      // Trigger analytics event
      await analytics.track({
        event: "user_signed_in",
        userId: response.user.id
      })
    }
    
    return ctx
  })
}]
```

### Example: Modify Response

```ts
after: [{
  matcher: (context) => {
    return context.path === "/session"
  },
  handler: createAuthMiddleware(async (ctx) => {
    const session = ctx.context.returned
    
    if (session) {
      // Add custom field to response
      return ctx.json({
        ...session,
        customField: "custom value"
      })
    }
    
    return ctx
  })
}]
```

## Hook Context

The context object available in hook matchers:

<ParamField path="path" type="string | undefined">
  The endpoint path being called
</ParamField>

<ParamField path="context" type="AuthContext">
  The authentication context with additional properties:
  
  - `returned?: unknown` - The value returned by the endpoint (after hooks only)
  - `responseHeaders?: Headers` - Response headers (after hooks only)
</ParamField>

<ParamField path="headers" type="Headers | undefined">
  Request headers
</ParamField>

<ParamField path="request" type="Request | undefined">
  The HTTP request object (only for API requests)
</ParamField>

<ParamField path="body" type="any">
  Request body (for POST requests)
</ParamField>

<ParamField path="query" type="Record<string, string>">
  Query parameters
</ParamField>

## Matcher Function

The matcher function determines when a hook should run:

```ts
matcher: (context: HookEndpointContext) => boolean
```

### Matching by Path

```ts
matcher: (context) => {
  return context.path === "/sign-in/email"
}
```

### Matching Multiple Paths

```ts
matcher: (context) => {
  const paths = ["/sign-in/email", "/sign-up/email", "/sign-in/username"]
  return paths.includes(context.path || "")
}
```

### Matching by Pattern

```ts
matcher: (context) => {
  return context.path?.startsWith("/oauth/") || false
}
```

### Matching by Header

```ts
matcher: (context) => {
  const apiKey = context.headers?.get("x-api-key")
  return apiKey === "expected-value"
}
```

### Complex Matching

```ts
matcher: (context) => {
  // Only match POST requests to sign-in endpoints
  const isSignIn = context.path?.includes("sign-in") || false
  const method = context.request?.method
  return isSignIn && method === "POST"
}
```

## Handler Function

The handler is an `AuthMiddleware` created with `createAuthMiddleware`:

```ts
import { createAuthMiddleware } from "better-auth/api"

handler: createAuthMiddleware(async (ctx) => {
  // Your logic here
  return { context: ctx }
})
```

### Handler Context

The handler receives a context object with:

<ResponseField name="request" type="Request">
  The HTTP request object
</ResponseField>

<ResponseField name="context" type="AuthContext">
  The full authentication context
</ResponseField>

<ResponseField name="body" type="any">
  Parsed request body
</ResponseField>

<ResponseField name="query" type="Record<string, string>">
  Query parameters
</ResponseField>

<ResponseField name="params" type="Record<string, string>">
  URL parameters
</ResponseField>

### Handler Return Values

**Return nothing** - Continue normal execution:

```ts
handler: createAuthMiddleware(async (ctx) => {
  console.log("Hook executed")
  // Implicit return of undefined
})
```

**Return modified context** - Update context for subsequent handlers:

```ts
handler: createAuthMiddleware(async (ctx) => {
  return {
    context: {
      ...ctx,
      customData: "value"
    }
  }
})
```

**Return response** - Stop execution and send response (after hooks):

```ts
handler: createAuthMiddleware(async (ctx) => {
  return ctx.json({
    message: "Custom response"
  })
})
```

## Helper Functions

### getSessionFromCtx

Get the current session from the context:

```ts
import { getSessionFromCtx } from "better-auth/api"

handler: createAuthMiddleware(async (ctx) => {
  const session = await getSessionFromCtx(ctx)
  
  if (!session) {
    throw new APIError("UNAUTHORIZED", {
      message: "Session required"
    })
  }
  
  console.log(`User: ${session.user.id}`)
  return { context: ctx }
})
```

### sessionMiddleware

Built-in middleware that adds session to context:

```ts
import { sessionMiddleware } from "better-auth/api"

endpoints: {
  protectedEndpoint: createAuthEndpoint("/protected", {
    method: "GET",
    use: [sessionMiddleware]
  }, async (ctx) => {
    // ctx.context.session is now available
    const session = ctx.context.session
    return ctx.json({ userId: session.user.id })
  })
}
```

## Complete Example

```ts
import { createAuthMiddleware, getSessionFromCtx } from "better-auth/api"
import type { BetterAuthPlugin } from "better-auth"

export const auditLogPlugin = () => {
  return {
    id: "audit-log",
    
    schema: {
      auditLog: {
        fields: {
          userId: {
            type: "string",
            references: {
              model: "user",
              field: "id",
              onDelete: "cascade"
            }
          },
          action: {
            type: "string"
          },
          timestamp: {
            type: "date"
          },
          metadata: {
            type: "string"
          }
        }
      }
    },
    
    hooks: {
      after: [
        {
          matcher: (context) => {
            // Log all sign-in and sign-up events
            const path = context.path || ""
            return path.includes("sign-in") || path.includes("sign-up")
          },
          handler: createAuthMiddleware(async (ctx) => {
            const session = await getSessionFromCtx(ctx)
            
            if (session) {
              await ctx.context.adapter.create({
                model: "auditLog",
                data: {
                  userId: session.user.id,
                  action: ctx.context.path || "unknown",
                  timestamp: new Date(),
                  metadata: JSON.stringify({
                    ip: ctx.request?.headers.get("x-forwarded-for"),
                    userAgent: ctx.request?.headers.get("user-agent")
                  })
                }
              })
            }
            
            return { context: ctx }
          })
        },
        {
          matcher: (context) => {
            return context.path === "/sign-out"
          },
          handler: createAuthMiddleware(async (ctx) => {
            console.log("User signed out")
            
            // Clean up resources
            // Send notification
            // etc.
            
            return { context: ctx }
          })
        }
      ]
    }
  } satisfies BetterAuthPlugin
}
```

## Best Practices

<Card title="Keep Matchers Simple" icon="bolt">
  Matcher functions should be fast and simple. Avoid complex logic or async operations.
</Card>

<Card title="Handle Errors Gracefully" icon="shield">
  Always handle errors in your handlers to prevent breaking the auth flow.
</Card>

<Card title="Avoid Heavy Operations" icon="gauge">
  Hooks run on every matching request. Keep operations lightweight and consider using background jobs for heavy tasks.
</Card>

<Card title="Type Safety" icon="code">
  Use TypeScript and the `satisfies` operator to ensure type safety.
</Card>
