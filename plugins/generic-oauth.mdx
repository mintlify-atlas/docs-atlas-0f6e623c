## Overview

The Generic OAuth plugin allows you to add custom OAuth 2.0 providers that aren't built into Better Auth. This is useful for:

- Internal OAuth providers (company SSO)
- Regional providers not in the core library
- OAuth providers with custom implementations
- Testing OAuth flows

## Installation

The Generic OAuth plugin is built into Better Auth. No additional installation required.

## Usage

Configure a custom OAuth provider using the `genericOAuth` plugin:

```typescript title="auth.ts"
import { betterAuth } from "better-auth";
import { genericOAuth } from "better-auth/plugins";

export const auth = betterAuth({
  database: // your database config,
  plugins: [
    genericOAuth({
      config: [
        {
          providerId: "custom",
          discoveryUrl: "https://provider.com/.well-known/openid-configuration",
          clientId: process.env.CUSTOM_CLIENT_ID,
          clientSecret: process.env.CUSTOM_CLIENT_SECRET,
          scopes: ["openid", "profile", "email"],
        }
      ]
    })
  ]
});
```

## Configuration

<ParamField path="config" type="OAuth2Provider[]" required>
  Array of OAuth provider configurations
</ParamField>

### OAuth2Provider Interface

<ParamField path="providerId" type="string" required>
  Unique identifier for the provider
</ParamField>

<ParamField path="discoveryUrl" type="string">
  OpenID Connect discovery endpoint URL. If provided, authorization and token URLs will be auto-discovered.
</ParamField>

<ParamField path="authorizationUrl" type="string">
  OAuth authorization endpoint (required if discoveryUrl is not provided)
</ParamField>

<ParamField path="tokenUrl" type="string">
  OAuth token endpoint (required if discoveryUrl is not provided)
</ParamField>

<ParamField path="userInfoUrl" type="string">
  User info endpoint (required if discoveryUrl is not provided)
</ParamField>

<ParamField path="clientId" type="string" required>
  OAuth client ID from the provider
</ParamField>

<ParamField path="clientSecret" type="string" required>
  OAuth client secret from the provider
</ParamField>

<ParamField path="scopes" type="string[]">
  OAuth scopes to request. Default: `["openid", "profile", "email"]`
</ParamField>

<ParamField path="pkce" type="boolean">
  Enable PKCE (Proof Key for Code Exchange). Default: `true`
</ParamField>

<ParamField path="redirectURI" type="string">
  Custom redirect URI. Default: `{baseURL}/callback/{providerId}`
</ParamField>

<ParamField path="getUserInfo" type="function">
  Custom function to fetch and map user information from the provider
  
  ```typescript
  async (tokens: OAuth2Tokens) => Promise<{
    id: string;
    email: string;
    name: string;
    image?: string;
    emailVerified?: boolean;
  }>
  ```
</ParamField>

## Examples

### Using OpenID Discovery

Most modern OAuth providers support OpenID Connect discovery:

```typescript
genericOAuth({
  config: [
    {
      providerId: "keycloak",
      discoveryUrl: "https://keycloak.example.com/realms/myrealm/.well-known/openid-configuration",
      clientId: process.env.KEYCLOAK_CLIENT_ID!,
      clientSecret: process.env.KEYCLOAK_CLIENT_SECRET!,
      scopes: ["openid", "profile", "email"],
    }
  ]
})
```

### Manual Configuration

For providers without discovery support:

```typescript
genericOAuth({
  config: [
    {
      providerId: "legacy-provider",
      authorizationUrl: "https://provider.com/oauth/authorize",
      tokenUrl: "https://provider.com/oauth/token",
      userInfoUrl: "https://provider.com/api/user",
      clientId: process.env.LEGACY_CLIENT_ID!,
      clientSecret: process.env.LEGACY_CLIENT_SECRET!,
      scopes: ["user:read", "user:email"],
      getUserInfo: async (tokens) => {
        const response = await fetch("https://provider.com/api/user", {
          headers: {
            Authorization: `Bearer ${tokens.accessToken}`,
          },
        });
        const data = await response.json();
        
        return {
          id: data.user_id,
          email: data.email_address,
          name: data.display_name,
          image: data.avatar_url,
          emailVerified: data.email_confirmed,
        };
      },
    }
  ]
})
```

### Multiple Custom Providers

You can configure multiple custom OAuth providers:

```typescript
genericOAuth({
  config: [
    {
      providerId: "internal-sso",
      discoveryUrl: "https://sso.company.com/.well-known/openid-configuration",
      clientId: process.env.INTERNAL_CLIENT_ID!,
      clientSecret: process.env.INTERNAL_CLIENT_SECRET!,
    },
    {
      providerId: "partner-oauth",
      authorizationUrl: "https://partner.com/oauth/authorize",
      tokenUrl: "https://partner.com/oauth/token",
      userInfoUrl: "https://partner.com/api/user",
      clientId: process.env.PARTNER_CLIENT_ID!,
      clientSecret: process.env.PARTNER_CLIENT_SECRET!,
    }
  ]
})
```

## Client Usage

Sign in with the custom provider from the client:

```typescript title="sign-in.tsx"
import { authClient } from "./auth-client";

async function signInWithCustomProvider() {
  await authClient.signIn.social({
    provider: "custom", // matches providerId
    callbackURL: "/dashboard",
  });
}
```

## User Info Mapping

The `getUserInfo` function allows you to map the provider's user data to Better Auth's user schema:

```typescript
getUserInfo: async (tokens) => {
  // Fetch user data from provider
  const response = await fetch(userInfoUrl, {
    headers: {
      Authorization: `Bearer ${tokens.accessToken}`,
      Accept: "application/json",
    },
  });
  
  const userData = await response.json();
  
  // Map to Better Auth user format
  return {
    id: userData.sub || userData.id,
    email: userData.email,
    name: userData.name || `${userData.first_name} ${userData.last_name}`,
    image: userData.picture || userData.avatar_url,
    emailVerified: userData.email_verified || false,
  };
}
```

## PKCE Support

PKCE (Proof Key for Code Exchange) is enabled by default for enhanced security:

```typescript
genericOAuth({
  config: [
    {
      providerId: "provider",
      // ... other config
      pkce: true, // default
    }
  ]
})
```

Disable PKCE if your provider doesn't support it:

```typescript
pkce: false
```

## Redirect URI

By default, the redirect URI is constructed as:

```
{baseURL}/callback/{providerId}
```

You can customize it:

```typescript
genericOAuth({
  config: [
    {
      providerId: "custom",
      redirectURI: "https://myapp.com/auth/callback/custom",
      // ... other config
    }
  ]
})
```

<Note>
Make sure the redirect URI is registered in your OAuth provider's configuration.
</Note>

## Provider Setup

When setting up your custom OAuth provider, configure:

1. **Redirect URI**: `{YOUR_APP_URL}/callback/{providerId}`
2. **Client ID and Secret**: Store in environment variables
3. **Scopes**: Request appropriate permissions
4. **Grant Types**: Enable "Authorization Code" grant

## Debugging

Enable debug logging to troubleshoot OAuth flows:

```typescript
export const auth = betterAuth({
  // ... config
  advanced: {
    debug: true,
  },
});
```

This will log:
- Authorization URL construction
- Token exchange requests
- User info fetching
- Error responses

## Common Issues

<Accordion title="Authorization fails with 'invalid_scope' error">
Check that the scopes you're requesting are supported by the provider. Remove any unsupported scopes.
</Accordion>

<Accordion title="Token exchange fails with 'invalid_grant'">
Verify that:
- The redirect URI matches exactly (including trailing slashes)
- The client ID and secret are correct
- The authorization code hasn't expired
</Accordion>

<Accordion title="User info fetching fails">
Ensure:
- The `userInfoUrl` is correct
- The access token has the required scopes
- Implement a custom `getUserInfo` function if the default doesn't work
</Accordion>
