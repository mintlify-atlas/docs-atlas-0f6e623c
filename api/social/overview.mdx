# Social Providers

Better Auth provides built-in support for 30+ OAuth 2.0 social authentication providers. Social providers enable users to sign in using their existing accounts from platforms like Google, GitHub, Apple, and more.

## How It Works

Social authentication in Better Auth follows the OAuth 2.0 authorization code flow:

<Steps>
### User initiates sign-in

The user clicks a "Sign in with [Provider]" button, triggering the OAuth flow.

### Authorization redirect

Better Auth redirects the user to the provider's authorization page where they grant permissions.

### Authorization code exchange

The provider redirects back with an authorization code, which Better Auth exchanges for access tokens.

### User profile retrieval

Better Auth uses the access token to fetch the user's profile information from the provider.

### Account creation or linking

Better Auth creates a new user account or links to an existing account based on the email address.
</Steps>

## Basic Configuration

Configure social providers in your Better Auth instance:

```ts
import { betterAuth } from "better-auth";
import { google, github, discord } from "better-auth/social-providers";

export const auth = betterAuth({
  socialProviders: {
    google: {
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    },
    github: {
      clientId: process.env.GITHUB_CLIENT_ID!,
      clientSecret: process.env.GITHUB_CLIENT_SECRET!,
    },
    discord: {
      clientId: process.env.DISCORD_CLIENT_ID!,
      clientSecret: process.env.DISCORD_CLIENT_SECRET!,
    },
  },
});
```

## Common Options

All social providers support these common configuration options:

<ParamField path="clientId" type="string" required>
  The client ID from your OAuth application registration.
</ParamField>

<ParamField path="clientSecret" type="string" required>
  The client secret from your OAuth application registration.
</ParamField>

<ParamField path="redirectURI" type="string">
  Custom redirect URI for the OAuth callback. Defaults to `{baseURL}/api/auth/callback/{provider}`.
</ParamField>

<ParamField path="scope" type="string[]">
  Additional OAuth scopes to request beyond the default scopes.
</ParamField>

<ParamField path="disableDefaultScope" type="boolean" default={false}>
  Remove the provider's default scopes. Use with `scope` to fully customize requested permissions.
</ParamField>

<ParamField path="enabled" type="boolean" default={true}>
  Enable or disable the provider. Useful for conditional provider availability.
</ParamField>

## Advanced Options

<ParamField path="mapProfileToUser" type="function">
  Custom function to map the provider's profile data to Better Auth user fields.

  ```ts
  mapProfileToUser: (profile) => ({
    name: `${profile.first_name} ${profile.last_name}`,
    // Add custom fields
    role: profile.is_admin ? 'admin' : 'user',
  })
  ```
</ParamField>

<ParamField path="getUserInfo" type="function">
  Override the default user info retrieval logic. Receives OAuth tokens and should return user data.

  ```ts
  getUserInfo: async (token) => {
    // Custom API call to fetch user data
    const response = await fetch('https://api.example.com/user', {
      headers: { Authorization: `Bearer ${token.accessToken}` },
    });
    const profile = await response.json();
    return {
      user: {
        id: profile.id,
        email: profile.email,
        name: profile.name,
        emailVerified: profile.verified,
      },
      data: profile,
    };
  }
  ```
</ParamField>

<ParamField path="disableImplicitSignUp" type="boolean" default={false}>
  Prevent automatic account creation on first sign-in. Users must be explicitly registered before they can sign in.
</ParamField>

<ParamField path="disableSignUp" type="boolean" default={false}>
  Completely disable new user registration through this provider.
</ParamField>

<ParamField path="overrideUserInfoOnSignIn" type="boolean" default={false}>
  Update user profile data from the provider on every sign-in, not just on account creation.
</ParamField>

## Client-Side Usage

Use the Better Auth client to trigger social sign-in:

```ts
import { createAuthClient } from "better-auth/client";

const authClient = createAuthClient();

// Sign in with a provider
await authClient.signIn.social({
  provider: "google",
  callbackURL: "/dashboard", // Where to redirect after sign-in
});
```

## Security Considerations

<Warning>
  **Never expose client secrets in client-side code.** All OAuth configuration should be server-side only.
</Warning>

<Note>
  **Redirect URI validation:** Ensure your redirect URIs are properly configured in both Better Auth and the provider's developer console. Mismatched URIs are the most common cause of OAuth errors.
</Note>

### PKCE (Proof Key for Code Exchange)

Better Auth automatically uses PKCE for providers that support it, adding an extra layer of security to the OAuth flow.

### State Parameter

Better Auth automatically generates and validates state parameters to prevent CSRF attacks during the OAuth flow.

## Token Management

Better Auth automatically handles OAuth token storage and refresh:

- **Access tokens** are stored securely and used for API calls
- **Refresh tokens** (when available) are used to obtain new access tokens
- **Token expiration** is tracked and handled automatically

Access stored tokens through the account data:

```ts
const session = await auth.api.getSession({ headers: request.headers });
if (session?.user) {
  // Access OAuth account data
  const accounts = await auth.api.getAccounts({ userId: session.user.id });
  const googleAccount = accounts.find(acc => acc.providerId === 'google');
  
  if (googleAccount?.accessToken) {
    // Use the access token for provider API calls
  }
}
```

## Troubleshooting

### "redirect_uri_mismatch" error

This error occurs when the redirect URI in your OAuth request doesn't match what's configured in the provider's developer console. Ensure:

1. The redirect URI in Better Auth matches the provider console
2. The URI includes the correct protocol (http vs https)
3. The URI doesn't have trailing slashes if the provider doesn't expect them

### "invalid_client" error

This typically means:

1. Your client ID or client secret is incorrect
2. The OAuth application is not properly configured
3. The application is disabled in the provider console

### Missing user information

If user data like email is missing:

1. Check that you're requesting the correct OAuth scopes
2. Verify the user granted the necessary permissions
3. Some providers require email verification before sharing email addresses
