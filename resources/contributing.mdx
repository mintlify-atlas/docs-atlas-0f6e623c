Thank you for your interest in contributing to Better Auth! This guide will help you get started with the contribution process.

## Code of Conduct

This project and everyone participating in it is governed by our [Code of Conduct](https://github.com/better-auth/better-auth/blob/main/CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.

## Project Structure

The Better Auth monorepo is organized as follows:

```

├── packages/
│   ├── better-auth/       # Core authentication library
│   ├── cli/               # Command-line interface tools
│   ├── expo/              # Expo integration
│   ├── electron/          # Electron integration
│   ├── stripe/            # Stripe payment integration
│   ├── sso/               # SSO plugin (SAML & OIDC)
│   ├── drizzle-adapter/   # Drizzle ORM adapter
│   ├── prisma-adapter/    # Prisma ORM adapter
│   ├── kysely-adapter/    # Kysely adapter
│   ├── mongo-adapter/     # MongoDB adapter
│   └── redis-storage/     # Redis secondary storage
├── docs/                  # Documentation website
├── demo/                  # Demo applications
│   ├── nextjs/           # Next.js full-featured demo
│   ├── expo/             # Expo mobile demo
│   ├── electron/         # Electron desktop demo
│   └── stateless/        # Stateless JWT demo
└── e2e/                   # End-to-end tests
```

## Development Guidelines

When contributing to Better Auth, please follow these guidelines:

- **Keep changes focused** - Large PRs are harder to review and unlikely to be accepted. We recommend opening an issue and discussing it with us first.
- **Type safety is paramount** - Ensure all code is type-safe and takes full advantage of TypeScript features.
- **Write clear code** - Use self-explanatory code. Comments should only be used when truly necessary.
- **Consistent API design** - Maintain a consistent and predictable API across all supported frameworks.
- **Follow existing conventions** - Follow the existing code style and patterns.
- **Aim for stability** - Avoid changes that would require users to run migrations or update their config.

## Getting Started

### 1. Fork and Clone

Fork the repository to your GitHub account:

```bash
git clone https://github.com/your-username/better-auth.git
cd better-auth
```

### 2. Install Node.js

This project uses [nvm](https://github.com/nvm-sh/nvm) to manage Node.js versions:

```bash
nvm install
nvm use
```

Alternatively, install the LTS version from [nodejs.org](https://nodejs.org/).

### 3. Install pnpm

This project uses [pnpm](https://pnpm.io/) managed via [corepack](https://github.com/nodejs/corepack):

```bash
corepack enable
```

Or install globally with npm:

```bash
npm install -g pnpm
```

### 4. Install Dependencies

```bash
pnpm install
```

### 5. Set Up Environment Variables

Copy the example environment file:

```bash
# Unix/Mac
cp -n ./docs/.env.example ./docs/.env

# Windows
copy .\docs\.env.example .\docs\.env
```

### 6. Build the Project

```bash
pnpm build
```

### 7. Run Documentation Locally

```bash
pnpm -F docs dev
```

The documentation will be available at [http://localhost:3000](http://localhost:3000).

## Development Workflow

### 1. Create a Branch

Create a new branch for your changes:

```bash
git checkout -b type/description
```

**Branch type prefixes:**
- `feat/` - New features
- `fix/` - Bug fixes
- `docs/` - Documentation changes
- `refactor/` - Code refactoring
- `test/` - Test-related changes
- `chore/` - Build process or tooling changes

**Examples:**
```bash
git checkout -b feat/oauth-provider
git checkout -b fix/session-expiration
git checkout -b docs/improve-setup-guide
```

### 2. Make Your Changes

Follow the code style guidelines and add tests for your changes.

### 3. Code Formatting with BiomeJS

We use [BiomeJS](https://biomejs.dev/) for code formatting and linting:

```bash
# Format all code
pnpm format

# Check for linting issues
pnpm lint

# Fix auto-fixable issues
pnpm lint:fix
```

### 4. Run Database Containers

For testing database adapters:

```bash
docker compose up -d
```

<Note>
  On macOS, the **mssql** container requires Rosetta emulation and at least 2GB of allocated memory. See [container requirements](https://learn.microsoft.com/en-us/sql/linux/quickstart-install-connect-docker).
</Note>

### 5. Run Tests

```bash
# Run all tests
pnpm test

# Run tests for a specific package
pnpm -F better-auth test

# Run specific test file
vitest path/to/test-file.test.ts

# Run tests matching pattern
vitest -t "pattern"
```

<Warning>
  Do not run `pnpm test` directly as it runs tests in all packages, which takes a long time. Use `vitest` with specific paths or patterns instead.
</Warning>

### 6. Commit Your Changes

Follow the [Conventional Commits](https://www.conventionalcommits.org/) format:

**For features and fixes (included in changelog):**

```bash
feat(organization): add support for nested organizations

fix(two-factor): resolve TOTP verification issue

feat: add support for custom error messages

fix: resolve memory leak in session handling
```

**For documentation:**

```bash
docs: improve authentication flow explanation

docs: fix typos in API reference
```

**For non-functional changes:**

```bash
chore(refactor): reorganize authentication middleware

chore: update dependencies to latest versions
```

### 7. Push Your Branch

```bash
git push origin your-branch-name
```

### 8. Open a Pull Request

Open a pull request against the **canary** branch (not main).

**PR Description Should Include:**
- Clear description of what changes you made and why
- Relevant context or background
- List of breaking changes or deprecations
- Screenshots for UI changes
- References to related issues or discussions

**Example PR template:**

```markdown
## Description
Adds support for nested organizations, allowing organizations to have child organizations.

## Changes
- Added `parentId` field to organization schema
- Implemented recursive organization fetching
- Added tests for nested organization scenarios

## Breaking Changes
None

## Related Issues
Closes #123
```

## Testing

All contributions must include appropriate tests:

- Write unit tests for new features
- Ensure all tests pass before submitting
- Update existing tests if behavior changes
- Follow existing test patterns
- Test across different environments when applicable

**Adding JSDoc comments for regression tests:**

If a test prevents regression of a specific GitHub issue, add a `@see` comment:

```typescript
/**
 * @see https://github.com/better-auth/better-auth/issues/123
 */
it("should handle the previously broken behavior", async () => {
  // test implementation
});
```

## Pull Request Process

1. **Create a draft PR early** - Facilitate discussion and get feedback
2. **Reference related issues** - Link to relevant issues in your PR description
3. **Ensure tests pass** - All CI checks must pass
4. **Update documentation** - Keep docs in sync with code changes
5. **Keep it focused** - One feature or bug fix per PR
6. **Be responsive** - Respond to code review feedback promptly
7. **Update CHANGELOG** - For user-facing changes

## Code Style

- Follow the existing code style
- Use TypeScript types and interfaces effectively
- Keep functions small and focused
- Use meaningful variable and function names
- Add comments only for complex logic
- Update relevant documentation when making API changes
- Follow BiomeJS formatting rules
- **Avoid using Classes** - Use functions and objects instead

**Example of good code style:**

```typescript
// Good: Small, focused function with clear types
export async function createSession(
  userId: string,
  options: SessionOptions
): Promise<Session> {
  const session = await db.session.create({
    userId,
    expiresAt: calculateExpiration(options.expiresIn)
  });
  return session;
}

// Avoid: Large class with many responsibilities
class SessionManager {
  // ... many methods
}
```

## Component-Specific Guidelines

### Core Library (`/packages/better-auth`)

- Keep the core library focused on essential authentication functionality
- Plugins in the core are generally made by core members
- If you have a plugin idea, consider open-sourcing it independently
- Ensure all public APIs have JSDoc comments
- Maintain backward compatibility (provide migration path if breaking)
- Follow existing patterns for error handling and logging

### Documentation (`/docs`)

- Keep documentation up-to-date with code changes
- Use clear, concise language
- Include code examples for common use cases
- Document breaking changes in the migration guide
- Follow the existing documentation structure
- Test code examples to ensure they work

### Adapters

- Follow the adapter interface exactly
- Include comprehensive tests
- Document any adapter-specific quirks
- Support common use cases for the target database

### Plugins

- Keep plugins focused on a single feature
- Make plugins composable
- Document configuration options clearly
- Include examples of usage
- Test plugin interactions with core features

## Types of Contributions

### Bug Fixes

1. Search existing issues to avoid duplicates
2. Create an issue describing the bug
3. Wait for confirmation or approval
4. Implement the fix with tests
5. Submit a PR referencing the issue

### New Features

1. Open an issue to discuss the feature
2. Wait for approval from maintainers
3. Implement the feature with tests and docs
4. Submit a PR with detailed description

### Documentation

- Fix typos, improve clarity, add examples
- No prior approval needed for minor improvements
- For major documentation changes, open an issue first

### Examples

- Add new framework integrations
- Showcase advanced use cases
- Include a comprehensive README
- Ensure the example runs without errors

For security-related issues, please email [security@better-auth.com](mailto:security@better-auth.com) instead of opening a public issue.

Include:
- Detailed description of the vulnerability
- Steps to reproduce
- Potential impact

See our [Security Policy](/resources/security) for more information.
