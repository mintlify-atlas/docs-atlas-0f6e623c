## Getting Started

<Accordion title="Auth client not working - Import path issues">
  When encountering `createAuthClient` related errors, make sure to have the correct import path as it varies based on environment.

  **For React client-side components:**
  
  ```ts title="component.tsx"
  import { createAuthClient } from "better-auth/react";
  ```

  **For server-side usage** (Next.js middleware, server actions, server components):
  
  ```ts title="server.ts"
  import { createAuthClient } from "better-auth/client";
  ```

  The React import provides hooks and client-only functionality, while the client import is designed for server environments.
</Accordion>

<Accordion title="getSession not working in server components">
  If you try to call `authClient.getSession` in a server environment (e.g., a Next.js server component), it doesn't work since it can't access the cookies directly.

  **Solution 1: Use the auth server instance**
  
  ```tsx title="server.tsx"
  import { auth } from "./auth";
  import { headers } from "next/headers";

  const session = await auth.api.getSession({
    headers: await headers()
  });
  ```

  **Solution 2: Pass headers to the client**
  
  ```tsx title="server.tsx"
  import { authClient } from "./auth-client";
  import { headers } from "next/headers";

  const session = await authClient.getSession({
    fetchOptions: {
      headers: await headers()
    }
  });
  ```
</Accordion>

<Accordion title="Difference between getSession and useSession">
  Both serve different purposes based on your use case:

  **`useSession` (React Hook)**
  - Triggers re-renders when session data changes
  - Use for UI components that need to react to session updates
  - Client-side only

  ```tsx
  const { data: session, isPending } = useSession();
  ```

  <Warning>
    For performance reasons, do not use this hook in your `layout.tsx` file. Instead, use React Server Components and fetch session data via `auth.api.getSession`.
  </Warning>

  **`getSession` (Promise)**
  - Returns a promise with session data
  - No automatic re-renders
  - Available on both server and client
  - Use for one-time session checks or server-side logic

  ```tsx
  const { data: session, error } = await authClient.getSession();
  ```
</Accordion>

## Configuration

<Accordion title="How to configure a corporate proxy for server-side requests">
  If you need to configure a corporate proxy for Better Auth's outbound requests (e.g., OAuth provider requests), you can use undici's ProxyAgent:

  ```typescript title="auth.ts"
  import { betterAuth } from "better-auth";
  import { ProxyAgent, setGlobalDispatcher } from "undici";

  const proxyAgent = new ProxyAgent("http://your-proxy.example.com:8080");
  setGlobalDispatcher(proxyAgent);

  export const auth = betterAuth({
    // your configuration
  });
  ```

  Since Better Auth runs server-side, you have full control over Node.js's fetch behavior through undici's global dispatcher.

  <Note>
    Learn more about undici ProxyAgent in the [undici documentation](https://github.com/nodejs/undici/blob/main/docs/docs/api/ProxyAgent.md).
  </Note>
</Accordion>

<Accordion title="Adding custom fields to the users table">
  Better Auth provides a type-safe way to extend the user and session schemas.

  See our documentation on [extending core schema](/docs/concepts/database#extending-core-schema) for detailed instructions.

  Example:
  ```typescript
  export const auth = betterAuth({
    // ... other config
    user: {
      additionalFields: {
        role: {
          type: "string",
          required: true,
          defaultValue: "user"
        },
        phoneNumber: {
          type: "string",
          required: false
        }
      }
    }
  });
  ```
</Accordion>

<Accordion title="Can I remove name, image, or email fields from the user table?">
  At this time, you cannot remove the `name`, `image`, or `email` fields from the user table. These are considered core fields required by Better Auth.

  We do plan to have more customizability in the future, but for now, these fields must remain in your schema.
  
  You can leave them as nullable or provide default values if they're not needed for your use case.
</Accordion>

## Troubleshooting

<Accordion title="Common TypeScript Errors">
  If you're facing TypeScript errors, ensure your `tsconfig.json` has strict mode enabled:

  ```json title="tsconfig.json"
  {
    "compilerOptions": {
      "strict": true
    }
  }
  ```

  If you can't set `strict` to `true`, enable `strictNullChecks` at minimum:

  ```json title="tsconfig.json"
  {
    "compilerOptions": {
      "strictNullChecks": true
    }
  }
  ```

  Learn more in our [TypeScript documentation](/docs/concepts/typescript#typescript-config).
</Accordion>

<Accordion title="Dual Module Hazard Issue">
  If you're encountering errors like:

  ```
  No request state found. Please make sure you are calling this function 
  within a `runWithRequestState` callback.
  ```

  This is typically caused by a **dual module hazard** - when multiple versions of `better-auth` or `@better-auth/core` exist in your dependency tree.

  ### Diagnose the Issue

  Check for duplicate versions:

  ```bash
  # pnpm
  pnpm why @better-auth/core
  pnpm why better-auth

  # npm
  npm ls @better-auth/core
  npm ls better-auth

  # yarn
  yarn why @better-auth/core
  yarn why better-auth
  ```

  ### Fix: Clean Reinstall

  ```bash
  # Remove node_modules and lockfile
  rm -rf node_modules
  rm pnpm-lock.yaml # or package-lock.json or yarn.lock

  # Reinstall dependencies
  pnpm install # or npm install or yarn install
  ```

  Ensure all Better Auth packages in `package.json` use compatible versions.

  ### Fix: Package Manager Resolutions

  For Yarn v1 or pnpm v9, force `better-call` to resolve to a single instance:

  ```json title="package.json"
  {
    "dependencies": {
      "better-call": "^1.1.8"
    },
    "resolutions": {
      "better-call": "^1.1.8"
    }
  }
  ```

  <Warning>
    Install `better-auth` and related packages in `dependencies`, **not** `devDependencies`.
  </Warning>

  ### Fix: Bundler Configuration

  **For Next.js:**
  
  ```ts title="next.config.js"
  const config = {
    serverExternalPackages: ['better-auth']
  };
  ```

  **For Cloudflare Workers:**
  
  ```toml title="wrangler.toml"
  compatibility_flags = ["nodejs_compat"]
  ```

  ### Verify the Fix

  ```bash
  pnpm why @better-auth/core
  pnpm why better-auth
  pnpm why better-call
  ```

  You should see only one version listed for each package.
</Accordion>

## Database & Adapters

<Accordion title="Which database should I use?">
  Better Auth supports multiple databases through adapters:

  - **Prisma** - Best for existing Prisma projects
  - **Drizzle** - Lightweight and type-safe
  - **Kysely** - SQL query builder
  - **MongoDB** - NoSQL option
  - **Memory** - For testing only

  See our [adapters documentation](/docs/adapters) for setup instructions.
</Accordion>

<Accordion title="How do I run database migrations?">
  Database migrations depend on your adapter:

  **Prisma:**
  ```bash
  npx prisma migrate dev
  ```

  **Drizzle:**
  ```bash
  npx drizzle-kit generate
  npx drizzle-kit migrate
  ```

  Better Auth will automatically create the necessary tables on first run if you don't run migrations manually.
</Accordion>

## Authentication Methods

<Accordion title="How do I add OAuth providers?">
  Configure OAuth providers in your Better Auth setup:

  ```typescript title="auth.ts"
  import { betterAuth } from "better-auth";

  export const auth = betterAuth({
    socialProviders: {
      github: {
        clientId: process.env.GITHUB_CLIENT_ID!,
        clientSecret: process.env.GITHUB_CLIENT_SECRET!
      },
      google: {
        clientId: process.env.GOOGLE_CLIENT_ID!,
        clientSecret: process.env.GOOGLE_CLIENT_SECRET!
      }
    }
  });
  ```

  See the [OAuth documentation](/docs/authentication/social) for provider-specific setup.
</Accordion>

<Accordion title="How do I implement two-factor authentication?">
  Enable the two-factor plugin:

  ```typescript title="auth.ts"
  import { betterAuth } from "better-auth";
  import { twoFactor } from "better-auth/plugins";

  export const auth = betterAuth({
    plugins: [twoFactor()]
  });
  ```

  See the [2FA documentation](/docs/plugins/2fa) for implementation details.
</Accordion>

## Performance & Optimization

<Accordion title="How can I improve session lookup performance?">
  Consider using a Redis secondary storage for sessions:

  ```typescript
  import { betterAuth } from "better-auth";
  import { redis } from "@better-auth/redis-storage";

  export const auth = betterAuth({
    secondaryStorage: redis({
      url: process.env.REDIS_URL
    })
  });
  ```

  This caches sessions in Redis, reducing database queries.
</Accordion>

<Accordion title="Rate limiting is blocking legitimate users">
  Adjust rate limit settings in your configuration:

  ```typescript
  export const auth = betterAuth({
    rateLimit: {
      window: 60, // seconds
      max: 100 // requests per window
    }
  });
  ```

  See [rate limiting documentation](/docs/concepts/rate-limit) for more options.
</Accordion>

## Deployment

<Accordion title="How do I deploy to Vercel/Netlify/Cloudflare?">
  Better Auth works with all major hosting providers. Key considerations:

  - Set environment variables in your hosting dashboard
  - Configure `baseURL` for production
  - Ensure database is accessible from your hosting environment
  - For Cloudflare Workers, enable `nodejs_compat` flag

  See platform-specific guides in our [deployment documentation](/docs/guides/deployment).
</Accordion>
