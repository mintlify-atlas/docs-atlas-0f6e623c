## How to Update

Update Better Auth to the latest version:

```bash
# npm
npm update better-auth

# pnpm
pnpm update better-auth

# yarn
yarn upgrade better-auth
```

### Update Specific Version

```bash
# npm
npm install better-auth@latest

# pnpm
pnpm add better-auth@latest

# yarn
yarn add better-auth@latest
```

## Release Channels

Better Auth uses different release channels:

- **Latest** (`latest`) - Stable releases for production use
- **Beta** (`beta`) - Pre-release versions with new features
- **Canary** (`canary`) - Bleeding edge, built from the main branch

### Installing Beta Releases

```bash
pnpm add better-auth@beta
```

### Installing Canary Releases

```bash
pnpm add better-auth@canary
```

<Warning>
  Beta and canary releases may contain breaking changes and are not recommended for production use.
</Warning>

## Breaking Changes

Better Auth follows [Semantic Versioning](https://semver.org/):

- **Major versions** (e.g., 1.0.0 → 2.0.0) - Breaking changes
- **Minor versions** (e.g., 1.0.0 → 1.1.0) - New features, backward compatible
- **Patch versions** (e.g., 1.0.0 → 1.0.1) - Bug fixes, backward compatible

Breaking changes are only introduced in major version releases and are clearly documented in release notes.

## Release Process

Better Auth follows this release process:

1. **Development** - Features and fixes merged to `canary` branch
2. **Beta Testing** - Pre-releases published with `beta` tag
3. **Release Candidate** - Final testing before stable release
4. **Stable Release** - Published with `latest` tag
5. **Patch Releases** - Bug fixes and security updates

## Versioning Policy

We aim for stability:

- **No breaking changes in minor/patch releases**
- **Clear migration guides for major versions**
- **Deprecation warnings before removal**
- **Security patches backported when possible**
